# 开放咖啡豆二维码标准 v1 设计

状态：v1 候选设计与参考实现验证通过，等待印刷样本验证

## 1. 目标

建立一套独立于“咖一杯”账号体系的开放咖啡豆二维码标准：

- 烘焙商通过独立发行平台创建咖啡豆批次并生成二维码。
- 咖一杯用户扫描二维码后，可以在离线状态下读取完整豆子资料并确认入库。
- 其他软件可以依据公开协议实现兼容的编码器和解码器。
- 发行平台负责发行方身份、签名和编码表发布，但不成为历史二维码解析的单点依赖。

## 2. 明确不做

v1 不承担以下职责：

- 不作为法律意义上的商品防伪系统。
- 不处理支付、订单、会员或供应链履约。
- 不在二维码中保存消费者身份、手机号等个人信息。
- 不要求烘焙商注册“咖一杯”普通用户账号。
- 不依赖在线批次详情才能完成基础入库。

## 3. 角色与组件

### 3.1 烘焙商

烘焙商是二维码的数据发行方。其账号属于独立发行平台，不属于咖一杯用户系统。

### 3.2 独立发行平台

负责：

- 发行方注册和 `issuer_id` 分配。
- 批次编辑、字段校验和 `batch_id` 生成。
- 使用服务端私钥完成数据签名。
- 输出 PNG、SVG 或 PDF 二维码。
- 发布由平台根密钥签名的发行方目录、公钥、状态和密钥轮换信息。

发行方私钥不得进入浏览器、小程序、二维码、Excel 或公开接口。

### 3.3 开放标准与编码表

负责定义：

- 协议版本、配置档标识和编码注册表版本。
- 字段编号、数据类型、长度和必填规则。
- `qr_id` 与公开主编码的稳定映射。
- 自定义文字扩展、签名、兼容和废弃规则。
- 可供第三方复现的测试向量。

### 3.4 消费端

咖一杯是第一款消费端。其他软件可以依据公开协议自行实现解析、验证和入库。

## 4. 总体数据流

### 4.1 生成

1. 烘焙商登录独立发行平台。
2. 选择国家、产区、庄园或处理厂、豆种和处理法。
3. 填写商品名、产季、烘焙日期、烘焙度、净含量、海拔和风味描述。
4. 未收录实体使用 `qr_id = 0` 并填写对应原始文字。
5. 服务端生成唯一 `batch_id` 和规范化数据包。
6. 服务端使用发行方当前签名密钥签名。
7. 将数据包编码为普通二维码并输出印刷文件。

### 4.2 扫描

1. 消费端扫描并读取二维码字符串。
2. 检查协议前缀、版本和长度上限。
3. Base45 解码并解析 COSE_Sign1 与确定性 CBOR。
4. 使用本地编码表将 `qr_id` 还原为主编码和显示名称。
5. 使用缓存或在线获取的发行方公钥验证签名。
6. 展示完整入库预览。
7. 用户确认后保存，同时保存 `batch_id`、二维码摘要和验证状态。

## 5. 二维码外层格式

v1 文本前缀：

```text
COF1:<BASE45_PAYLOAD>
```

`COF1` 表示 Coffee Open Format v1。二维码必须使用普通 QR Code，而不是用微信小程序码的 `scene` 承载完整数据。

v1 数据编码顺序：

```text
业务字段 -> 确定性 CBOR -> COSE_Sign1 / Ed25519 -> Base45 -> COF1: 前缀
```

v1 候选版暂不固定 DEFLATE，避免不同实现对封装方式和参数理解不一致。完成至少 100 组真实豆单及 20-30 mm 印刷测试后，只有压缩能稳定降低二维码等级并保持识别率时，才在后续固定配置中统一加入；不得允许每条二维码自行选择压缩参数。

## 6. 全局 qr_id

### 6.1 规则

- 国家、产区、庄园、处理厂、豆种、处理法等实体共用全局唯一 `qr_id`。
- `qr_id` 为全局递增的无符号 32 位整数，CBOR 按最短形式编码。
- `qr_id = 0` 永久保留，表示未收录实体，需要读取原始文字字段。
- 已分配的 `qr_id` 永不修改、删除、回收或重新指向其他实体。
- 实体停用后保留原映射，通过 `replacement_qr_id` 指向替代实体。
- 现有 `CO-ET`、`RG-EA-YIR` 等公开主编码继续作为可读主键。
- `qr_id` 不编码国家、实体类型或层级，归属关系只由版本化编码表维护。
- 第三方不得自行分配正式 `qr_id`。

### 6.2 索引字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `qr_id` | uint | 全局永久编号 |
| `canonical_code` | text | 现有公开主编码 |
| `entity_type` | enum | 国家、产区、庄园等 |
| `introduced_version` | text | 首次发布版本 |
| `status` | enum | active、deprecated、retired |
| `retired_version` | text? | 停用版本 |
| `replacement_qr_id` | uint? | 推荐替代实体 |
| `record_hash` | text | 记录内容摘要 |
| `notes` | text? | 维护说明 |

## 7. v1 数据字段

字段采用固定数字标签，避免在二维码中重复长字段名。具体标签值以“二维码字段定义”表为准。

### 7.1 协议字段

| 字段 | 必填 | 说明 |
|---|---|---|
| `schema_version` | 是 | 协议版本 |
| `profile_id` | 是 | v1 数据配置档 |
| `registry_id` | 是 | 编码注册表标识 |
| `registry_version` | 是 | 不可变编码表快照版本 |
| `registry_hash` | 是 | 对应编码表快照摘要 |

### 7.2 发行字段

| 字段 | 必填 | 说明 |
|---|---|---|
| `issuer_id` | 是 | 独立发行方编号 |
| `issuer_name` | 是 | 离线显示用烘焙商品牌简称 |
| `key_id` | 是 | 签名密钥编号 |
| `batch_id` | 是 | 发行方范围内唯一批次编号 |
| `issued_at` | 是 | 生成时间 |

### 7.3 产地字段

| 字段 | 必填 | 说明 |
|---|---|---|
| `country_qr_id` | 是 | 国家 |
| `entity_roles` | 否 | 带角色的产区、庄园、生产者、水洗站、处理厂等实体数组 |
| `variety_qr_ids` | 否 | 豆种，可多值 |
| `process_qr_ids` | 是 | 处理法，可多值以表达组合处理 |

`entity_roles` 中每个实体必须注明 `country`、`region`、`farm`、`producer`、`washing_station` 或 `mill` 等角色。任何实体值为 0 时，必须同时提供原始文字和语言代码。

### 7.4 批次字段

| 字段 | 必填 | 限制 |
|---|---|---|
| `product_name` | 是 | 1-60 个 Unicode 字符 |
| `harvest_year` | 否 | 四位年份或跨年文本 |
| `roast_date` | 是 | ISO 8601 日期 |
| `roast_level` | 是 | 标准编码或受控枚举 |
| `net_weight_g` | 否 | 正整数，单位克 |
| `altitude_min_m` | 否 | 整数，单位米 |
| `altitude_max_m` | 否 | 整数，单位米 |
| `flavor_tag_qr_ids` | 否 | 标准风味标签，可多值 |
| `flavor_notes` | 是 | 最多 60 个 Unicode 字符，离线完整携带 |
| `lot_text` | 否 | 最多 40 个 Unicode 字符 |
| `language` | 是 | 文本字段默认语言，BCP 47 标签 |
| `custom_fields` | 否 | 受长度限制的扩展字段 |

字段状态必须区分：缺失表示未提供，显式 `unknown` 表示发行方确认未知，显式 `not_applicable` 表示该字段不适用。数组最大 8 项，单个文本最大 60 个 Unicode 字符，解码后业务数据总长度最大 4096 字节。

## 8. 签名与验证状态

v1 固定使用 COSE_Sign1 与 Ed25519。签名对象是确定性 CBOR 编码后的完整业务字段，不允许实现方自行拼接签名字节。

签名状态：

- `verified`：签名有效，公钥与 `issuer_id`、`key_id` 匹配。
- `pending`：结构有效，但离线或尚未取得发行方公钥。允许进入用户私人豆仓，联网后自动补验。
- `invalid`：签名不匹配、数据被修改或结构不合法。禁止自动入库。

发行方状态单独记录为 `authenticated`、`registered`、`revoked` 或 `unknown`。签名有效只证明数据由对应密钥签发，不证明烘焙商真实身份、豆子信息正确或商品真伪。

发行方目录必须由平台根密钥签名，消费端内置平台根公钥。目录记录密钥启用、轮换、撤销和泄漏时间。发行方被暂停或撤销不等同于历史签名失效，消费端应保留签发时间和撤销时间以便审计。

`pending` 数据不得进入公共编码表、公开榜单或作为认证资料传播。若后续补验变为 `invalid`，不自动删除用户记录，而是移除来源认证、醒目标记异常，并允许用户转为手工记录。

## 9. 重复与幂等

消费端以以下组合识别同一批次：

```text
issuer_id + batch_id
```

同时保存完整二维码数据摘要，用于发现同一批次编号对应不同内容的异常情况。

重复扫描时：

- 内容摘要相同：提示该批次已经入库。
- 内容摘要不同：标记冲突，禁止覆盖原记录，要求用户查看差异。

## 10. 版本与兼容

- 协议版本与编码表版本分开管理。
- 编码表以 `registry_id + registry_version + registry_hash` 唯一确定不可变快照。
- 编码表快照必须公开下载并允许第三方实现使用。
- 旧版消费端遇到未知可选字段时必须忽略并保留原始数据。
- 未知必填字段或不支持的主版本必须停止自动入库。
- 编码表更新只能新增、停用或建立替代关系。
- 每次编码表发布必须包含版本号、发布时间、内容摘要和增量变更记录。
- 第三方实现不得根据显示名称推测 `qr_id`。

## 11. 编码表工作簿扩展

在现有工作簿中增加：

1. `二维码实体索引`
2. `二维码字段定义`
3. `二维码版本表`
4. `烘焙批次模板`
5. `二维码测试样例`

现有“总编码表”和各分类表不改变主编码含义。首版 `qr_id` 按稳定排序一次性分配，后续只能从当前最大值继续追加。

## 12. 错误处理

| 场景 | 行为 |
|---|---|
| 非 `COF1` 二维码 | 提示不是支持的咖啡豆二维码 |
| Base45、COSE 或 CBOR 结构损坏 | 停止解析 |
| 不支持的主版本 | 提示更新应用或使用兼容解析服务 |
| 编码表版本较新 | 使用二维码自带回退文字预览，联网获取对应快照 |
| 公钥暂不可用 | 标记 pending，允许确认入库 |
| 签名明确无效 | 标记 invalid，禁止自动入库 |
| pending 后补验为 invalid | 保留私人记录，移除认证并标记异常 |
| 未知实体且有原文 | 保存原文并等待映射 |
| 未知实体且无原文 | 阻止完整入库 |
| 批次重复 | 提示已入库 |
| 批次编号冲突 | 禁止覆盖并展示差异 |

## 13. 测试要求

公开标准必须提供固定测试向量：

- 最小必填字段二维码。
- 包含全部可选字段的二维码。
- 多豆种、多处理站二维码。
- 包含 60 个中文字符风味描述的边界二维码。
- 未收录庄园的 `qr_id = 0` 回退二维码。
- 未知可选字段的前向兼容二维码。
- 正确签名、pending、错误签名和篡改数据样例。
- 编码表旧版本、实体停用和替代映射样例。
- 重复批次和批次编号冲突样例。
- 缺失、unknown、not_applicable 三种字段状态样例。
- 超长文本、超大数组和解码后超过 4096 字节的拒绝样例。

每个测试向量必须包含：原始业务字段、规范化 CBOR、最终二维码字符串、数据摘要、签名、公钥和预期解析结果。

## 14. 非功能要求

- 已安装本地编码表时，普通二维码解析目标时间小于 1 秒。
- 发行平台短时不可用不影响已生成二维码的基础解析。
- 消费端不得因 pending 状态将数据显示为“已认证”。
- 发行方密钥支持轮换，同一发行方可以存在多个历史 `key_id`。
- 公开协议、编码表快照和测试向量必须可独立下载和长期归档。
- 第一阶段按数百家发行方、数百万批次设计，但不为超大规模提前引入复杂分布式架构。
- “完整离线”指二维码携带完整批次数据，并且解析端持有对应编码表快照；未知 `qr_id` 必须有原始文字回退。
- 发布 v1 前必须使用至少 100 组真实豆单，在 20、25、30 mm 印刷尺寸上验证字符容量和识别率。

## 15. 决策记录

| 编号 | 决策 | 备选 | 原因 |
|---|---|---|---|
| D-001 | 烘焙商生成二维码，用户扫描入库 | 用户自行生成 | 数据源更接近原始信息 |
| D-002 | 发行账号独立于咖一杯 | 复用咖一杯账号 | 降低平台绑定感，便于行业接入 |
| D-003 | 二维码协议公开 | 仅咖一杯可读 | 形成可扩展标准，降低烘焙商顾虑 |
| D-004 | 二维码自身携带完整数据 | 仅保存在线短编号 | 支持离线和长期可读 |
| D-005 | 中央服务只管理发行身份和版本 | 所有批次必须在线查询 | 避免单点依赖 |
| D-006 | 使用全局唯一 `qr_id` | 各实体类型分别编号 | 解析和维护更简单，冲突风险更低 |
| D-007 | 风味描述离线完整携带，最多 60 字符 | 详细描述在线读取 | 满足完整入库 |
| D-008 | pending 状态允许入库并后续补验 | 必须联网验证后入库 | 兼顾离线可用性与来源验证 |
| D-009 | 明确无效签名禁止自动入库 | 仅警告仍允许入库 | 防止篡改数据被当作烘焙商资料 |
| D-010 | v1 不作为防伪系统 | 同时承担商品防伪 | 控制第一版范围和法律风险 |
| D-011 | 使用确定性 CBOR + COSE_Sign1 / Ed25519 | 自定义签名字节格式 | 采用成熟开放标准，减少跨实现歧义 |
| D-012 | v1 候选版暂不固定 DEFLATE | 每条二维码可选压缩 | 先用真实印刷样本验证收益，避免实现分叉 |
| D-013 | 签名状态与发行方状态分离 | 单一可信状态 | 避免把密钥签名误解为身份认证或商品防伪 |
| D-014 | 编码表发布不可变且带注册表摘要 | 只携带版本文字 | 允许第三方准确定位和验证编码表快照 |
| D-015 | pending 只进入私人豆仓 | pending 与 verified 同等使用 | 保留离线能力，同时限制未验证数据传播 |

## 16. 第一阶段交付边界

第一阶段只交付：

- 扩展后的编码工作簿。
- 公开协议文档。
- 参考编码器和解码器。
- 固定测试向量。
- 咖一杯侧的离线解析能力验证。

独立发行平台的正式账号、审核和部署在参考协议验证通过后单独立项。

## 17. 参考实现验证结果

2026-07-12 已完成第一组可运行验证：

- 从源工作簿生成 854 个全局唯一 `qr_id`。
- 注册表包含 996 条关系和 1976 条别名。
- 注册表摘要为 `sha256:2378f3735e86959fe95f2a2735c46f9a53eb25f0476ed7b0a32d9f6c8fe3c4bc`。
- 样例完成确定性 CBOR、COSE_Sign1 / Ed25519、Base45 编码和签名验证。
- 已验证 `verified`、`pending`、`invalid` 三种签名状态。
- 已验证 `qr_id = 0` 缺少文字回退时会被拒绝。
- 已将二维码渲染为 PNG，并从像素中重新识别、解析和验证通过。
- 当前完整样例为 643 个字符，二维码 Version 16、81 x 81 模块、纠错等级 M。

当前结论：电子图片识别闭环可行；20 mm 印刷尺寸可能过密，必须继续完成 20、25、30 mm 的实际打印与不同手机扫码测试，才能决定是否需要固定压缩或进一步缩短字段。
